# Judge ルブリック（突き合わせ → 統合）

メイン Opus 4.8 が全パネリストの回答を以下の軸で構造化する。**平均や多数決ではない**——食い違いと盲点を能動的に洗い出すのが目的。

## 分析軸

| 軸 | 見るもの |
|---|---|
| **Consensus** | 複数パネリストが一致している点。確度が高い土台 |
| **Contradictions** | 食い違い。**どちらがより確からしいか**を根拠（出典・論理）つきで判定する。安易に折衷しない |
| **Partial coverage** | 一部のパネリストしか触れていない論点。抜けの補完 |
| **Unique insights** | 単独パネリストだけが出した有用な洞察。捨てずに拾う |
| **Blind spots** | 全員が見落としていそうな点。judge が能動的に補う（下の「系統的に漏れるカテゴリ」で機械的に点検する） |

- 各項目は**どのパネリスト由来か**を明示する（追跡可能性）。
- 矛盾の解決を最優先する。「両論併記で逃げる」のは最後の手段。

## Blind spots の能動チェック（系統的に漏れるカテゴリ）

パネリストは「本体の良し悪し」には強いが、**要素を分割・統合・連携させたときの“継ぎ目（seam）”を全員そろって落とす**傾向がある。最終回答を書く前に、以下を1つずつ問うて、抜けていれば Blind spots として補う。**設問が技術以外でも、対応する一般カテゴリとして適用する。**

| カテゴリ | 問い | 技術設問での代表例（漏れやすい） |
|---|---|---|
| **境界の検証** | 分割した要素が「約束（インターフェース/契約）」を本当に守ると、どう保証するのか？ | **契約テスト（contract testing）**、インターフェース回帰、互換性検証 |
| **境界をまたぐ整合性・原子性** | 複数要素にまたがる更新が途中で壊れない保証は？ | **Outbox パターン**、冪等性、Saga＋補償、二重書き込み期間 |
| **失敗モード** | 部分失敗・遅延・タイムアウト時にどうなるか？ | リトライ、サーキットブレーカー、デッドレター |
| **観測・追跡** | 問題が起きたとき、継ぎ目をまたいで原因を追えるか？ | 相関ID伝播、分散トレーシング、統一ログ |
| **移行の途中状態** | 「移行中」の半端な状態のコスト/リスクを誰か見積もったか？ | 二重運用コスト、切り戻し（rollback）、段階リリース |
| **コスト・撤退** | 固定費・運用人員・やめる時のコストは？ | 機会損失、ベンダーロックイン |
| **暗黙の前提** | 設問やパネリストが無自覚に置いている前提は妥当か？ | 「将来スケールするから」等の未検証前提 |

> 由来: 2026-06-15 の実走で、3パネリスト全員が **契約テスト**と **Outbox（境界をまたぐ整合性）**を落とした。個別トピックではなく「継ぎ目カテゴリ」を点検対象として一般化した。

## 最終出力の形

1. **最終回答**（本体・トップ）
   - 上の分析を根拠に、judge が一貫した1つの答えを書く。
2. **監査証跡（audit trail）**（畳んで添付）
   - Consensus / Contradictions / Partial / Unique / Blind spots を、パネリスト帰属つきで簡潔に。
   - **継ぎ目チェック表を必ず常設出力する**（下記）。
   - 読み手が「なぜこの結論か」を後から検証できる粒度にする。

### 継ぎ目チェック表（audit trail に必須）

「Blind spots の能動チェック」の全カテゴリを**1行も省略せず**表で出す。各カテゴリについて被覆状況を1つ選び、所見を書く：

- **✅ 被覆** … パネリストが十分に扱った（誰が、を明示）
- **△ 部分** … 触れたが浅い／一部のみ（何が足りないか）
- **🕳️ 欠落** … 全員が落とした → judge が補った内容を必ず書く
- **N/A** … この設問には当てはまらない（理由を一言）

| 継ぎ目カテゴリ | 判定 | 所見（帰属つき／欠落なら judge の補足） |
|---|---|---|
| 境界の検証 | | |
| 境界をまたぐ整合性・原子性 | | |
| 失敗モード | | |
| 観測・追跡 | | |
| 移行の途中状態 | | |
| コスト・撤退 | | |
| 暗黙の前提 | | |

> この表は「埋まっていること自体」が品質ゲート。空欄や行の省略は不可（N/A も理由を書く）。🕳️ 欠落が1つでもあれば、最終回答にその補足を反映する。

## judge の心構え

- パネリストの一致＝正しさ、ではない。**全員が同じ間違いをすることもある**（共通バイアス）。Blind spots で能動的に疑う。
- 特に「継ぎ目（境界の検証・整合性・失敗モード）」は全員そろって落ちやすい。上のチェックを必ず一巡する。
- 出典・再現性・論理の強さで重みづけする。多数決に流されない。
