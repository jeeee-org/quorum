# なぜ「独立並列」なのか

## 多様性は演出せず収穫する

役割（ペルソナ）を割り当てて多様性を作る方式（「あなたは批評家」「あなたは楽観派」）は、中身の伴わない見せかけの多様性になりやすい。

本スキルは逆に、**同じプロンプトを互いに見えない状態で独立に走らせる**。それだけで各回答者は異なる推論経路・異なるツール呼び出し・異なる参照ソースに分岐する。この「自然なブレ」を統合する方が、作り込んだ多様性より良い結果になる。

## 同一モデルの2回でも効く

別モデルが無くても成立する。**ホストの同じネイティブモデルを2回コールドに走らせて統合する**だけでも、単発の取りこぼしを補える。Claude Codeでは opus、Codexでは `codex-native` の独立サブエージェントが補完枠になる。異種モデルは多様性をさらに広げるための上乗せ。

## 欠けた枠はホストのネイティブ実行で補完する（バックフィル）

この性質を使い、**目標パネル数（既定3）を、まず利用可能な異種backend、足りなければホストのネイティブ実行で埋める**。Claude Codeホストの既定は **opus / codex / grok**、Codexホストの既定は **codex-native / claude / grok**。現在ホストと同名の外部runnerは除外し、CLIのネストとquorum再帰を防ぐ。別モデルが入る枠ほど多様性は高いが、同族補完でも偶発的な取りこぼしを減らせる。補完は `detect_panel.sh` が `QUORUM_HOST` と `QUORUM_PANEL_SIZE` に従って行い、特定バックエンドの増員・固定は `QUORUM_PANEL` の明示指定で行う。

**opus 枠の fable 差し替え（`QUORUM_NATIVE=fable`）はユーザーの呼びかけ時のみ**。judge と同格のモデルをパネル（幅）に使うのは「深さは judge・幅は安価なモデル」の基本配置に反し費用対効果が悪いが、最重要の1問で幅の質も上げたい局面のための明示オプション。補完で fable を増殖させることはない（補完チェーンに fable は入れない）。

## judge は最強モデルに座らせる（深さは judge・幅はパネル）

融合の品質上限は judge の突き合わせ能力で決まる——矛盾の裁定・重みづけ・盲点の補完は judge の知性がボトルネックになる。∴ **judge にはセッションで使える最も賢いモデルを座らせ、パネリストはネイティブ独立実行＋異種ベンダーで幅を稼ぐ**のが基本配置。

同一モデルの複数独立実行が拾えるのは「たまたまの取りこぼし」であって、モデル自身の系統的な盲点は N 体走らせても全員が同じ場所で転ぶ。**メインホストとは別ベンダーのモデルは訓練データも検索経路も癖も違うため、この相関しない誤りを相殺できる**——これはメインのモデルがどれだけ賢くなっても消えない、本スキルの中核価値。

## ブラインドを守る

パネリストどうしに中間結果を見せない。見せ合うと互いに引きずられ（アンカリング）、独立性＝多様性の源泉が潰れる。突き合わせは最後に judge が一括で行う。
